post 05 — found it
inode
internal/adapters/db/sqlite.go
found it.
the `SearchSimilar` query — read it carefully.
SELECT n.id, n.content_enc, n.summary,        n.category, n.tags, n.created_at FROM note_embeddings e JOIN notes n ON n.id = e.note_id WHERE e.embedding MATCH ? AND k = ? ORDER BY distance -- ← computed
↳
ordered by distance.
but never SELECTed.
sqlite-vec returned the relevance score for every match.
the adapter threw it away before returning to core.
consequence — no relevance floor. the K nearest are returned no matter how poor.

post 07 — what's L2 distance?
inode
a quick aside —
distance, demystified
voyage + ollama embeddings are L2-normalised,
so the distance between any two vectors lives in [0, 2].
identical
strong
moderate
unrelated
opposite
0.0
0.5
1.0
1.4
2.0
our cutoff
cheat sheet —
L2 = √( 2 · (1 − cos_sim) ) for unit vectors
d = 0.0 ↔ cos_sim = 1.00   → same direction
d = 1.0 ↔ cos_sim = 0.50   → ~50% aligned
d = 1.4 ↔ cos_sim ≈ 0   → orthogonal

post 08 — the fix
inode
3 small steps
the fix —
1.
surface the score
add e.distance to the SELECT, plumb to model.Note.
@@ db/sqlite.go @@ - SELECT n.id, n.content_enc, ... + SELECT e.distance, n.id, n.content_enc, ...
2.
filter at the core layer
drop notes where distance > threshold  (default 1.0).
@@ core/search.go @@ + notes = filterByDistance(notes, s.thresholdFor(opts)) if len(notes) == 0 { return noMatch() }
3.
make it tunable
new [search] block in ~/.inode/config.toml.
@@ ~/.inode/config.toml @@ + [search] + max_distance = 1.0 # 0 = built-in default · <0 = disable + top_k = 5

post 10 — local-first IS the product
inode
why we said no — for now
local-first is the product.
a personal knowledge tool that needs you to run a database server
isn't a personal knowledge tool. it's infrastructure.
keep SQLite when —
✓ single user, single device
✓ single binary, zero deps
✓ works offline (with ollama)
✓ < 10k notes / user
move to Postgres when —
→ multi-user / shared vaults
→ cross-device sync
→ hosted SaaS mode
→ concurrent writes at scale
the real reason —
db.Adapter is already an interface.
there is no lock-in cost to delaying the swap
until the use case actually arrives.
